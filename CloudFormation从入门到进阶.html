CloudFormation 从入门到进阶

AWS AWS CloudFormation 2022/04/23

AWS 的 CloudFormation 是非常好用的一个服务，这里介绍一下常见的使用方法。

1、CloudFormation简介

1.1、AWS Console操作遇到的问题

要说需要CloudFormation这个技术来解决什么问题，我们首先看下以往的AWS Console中遇到的一些问题。

无法复制粘贴-AWS Console是图形化操作的，所以你在测试账号里面创建VPC，子网，路由这些操作，在迁移到生产环境的时候，还需要重复的操作一遍。不仅浪费时间，还很容易出错。

无法批量操作-在AWS Console里面，你只能同时操作单个资源，例如当你清空一个测试VPC的时候，你一步步的删除实例、删除安全组、删除IGW，最后在删除VPC。如果不小心忘记删除了EIP，还会产生费用。

没有版本控制-很有可能运维A前天修改了一个安全组，运维B昨天修改了一个路由，今天业务不通了。你很难找到对这些内容的修改，CloudTrail的记录也并不直观。

CloudFormation可以很好的解决上面的三个问题。

1.2、CloudFormation模板

要使用CloudFormation，你需要先编写CloudFormation模板，CloudFormation 模板是 JSON 或 YAML 格式的文本文件，你可以指定任何的扩展名称，例如.txt。

下面是CloudFormation模板里面的内容，分别用JSON和YAML格式表示创建一个VPC的资源。JSON和YAML格式是可以相互转换的，但是YAML支持注释功能，所以我更习惯使用YAML格式来写模板，之后的CloudFormation模板的代码我都会使用YAML格式来写。

上面是通过模板创建了一个VPC资源，当然可以在模板里面定义更多的资源，例如创建安全组和EC2。那么这个模板里面所有的资源可以作为一个“堆栈”进行管理，可以通过AWS控制台，AWS CLI和API创建创建、更新、删除堆栈。

堆栈里面的资源可以进行更新，在更改资源之前，你可以生成一个更改集，里面包含可能会对运行资源造成的影响。你可以作出变更计划之后再更新堆栈。

利用CloudFormation，可以解决上面提到AWS Console操作会遇到的问题：

1、简化基础设施管理

你可以轻松删除堆栈，这将删除堆栈中的所有资源。通过使用 CloudFormation，你可以轻松地将一组资源作为一个单元进行管理。

2、快速复制您的基础设施

CloudFormation是一个文本文件，你可以非常方便的复制粘贴它。例如在测试账号测试完成后，快速的迁移到生产账号中，或者其他区域中。

3、轻松控制和跟踪对您的基础设施所做的更改

有些情况下你需要进行一系列操作，但是操作之后可能又需要回滚配置。因为CloudFormation是文本文件，所以可以像代码管理一样，利用版本控制来管理CloudFormation的模板文件，了解所做的更改。并且进行回滚操作。

1.3、CloudFormation 模板格式

前面提到过模板支持JSON和YAML两种格式，YAML格式可以添加注释，所以这里看下YAML格式的模板，最外层的所有参数。这里只会介绍几个重要的参数，其他参考大家可以自己看文档模板剖析。

Resources：这是唯一的必选字段，也就是模板里面其他字段都可以不填写，但是必须定义资源字段。

Outputs：这也是比较常用的字段，可以指定堆栈的输出信息，例如通过Outputs输出EC2的公有IP地址，这个输出可以被其他堆栈调用。也可以通过aws cloudformation describe-stacks查看输出的内容。

Mappings：可用来指定一些键值对。后续利用Fn::FindInMap内部函数调用这些键值对。

Parameters：如果你的模板想要给其他人使用，往往需要这个参数。例如不同账号的SSH Key并不相同，所以可以让用户自己指定Key，后续EC2的SSH Key调用这个变量即可。

1.4、CloudFormation 资源属性

要了解Resources字段的属性，主要还是明确需求之后去看文档AWS 资源和属性类型参考。

资源类型标识符总是采用这个格式：service-provider::service-name::data-type-name。

例如创建一个VPC：AWS::EC2::VPC

创建IGW并且关联到VPC：AWS::EC2::InternetGateway

在VPC里面创建子网：AWS::EC2::Subnet

这里子网需要指定VpcId，可以使用Ref内部函数来获取这个信息。Ref函数可以返回指定的参数或资源的值。例如你需要查看AWS::EC2::VPC就可以知道，它返回的是VpcId信息。

Ref有短格式的写法，可以写在一行!Ref logicalName

在VPC内创建路由表并关联到子网，路由表设置默认路由指向IGW：AWS::EC2::RouteTable、AWS::EC2::Route、AWS::EC2::SubnetRouteTableAssociation

在VPC内创建一个安全组：AWS::EC2::SecurityGroup

安全组里面数字-1表示允许所有。

创建一个EC2实例：AWS::EC2::Instance

1.5、AWS控制台创建堆栈

有了上面的一些知识，我们可以创建一个简单的堆栈来实验一下。

这个模板将上面的资源都整合起来了，模板内部：创建了一个VPC，VPC内部有个IGW，有默认路由指向了IGW。最后创建了一个EC2放置在公有子网里面，允许获取公网IP地址。

可以通过AWS CloudFormation控制台，API，或者AWS CLI这三种方式可以创建堆栈。

进入AWS CloudFormation控制台，点击创建堆栈，选择使用新资源创建。

选择模板已就绪，上传准备的yaml文件，文件最大支持1MB，文件的扩展名是txt并没有关系，AWS会自动创建一个S3桶来存储模板文件。你也可以将模板文件上传到上s3，然后指定s3的路径来调用模板。上传成功后你可以在利用CloudFormation Designer来图形化的查看堆栈资源信息。

通过Designer你可以以图形化的方式来创建、查看和修改模板。一般我个人更习惯使用YAML创建模板，然后使用Designer校验一下，并不会直接使用Designer来创建模板。

最后指定堆栈的名称，注意堆栈名称不能有下划线，可以有中横线。（资源名称不能有中横线，AWS好多地方命名规则并不统一。）

通过【事件】可以查看堆栈创建的进度，如果堆栈创建失败，通过事件输出可以非常容易的定位错误。

例如这里我故意将子网的AZ信息写成cn-northwest-1，少写了一个a。可以通过【事件】来查看报错信息，状态原因非常明确的说明了错误原因，你可以快速的根据“逻辑ID”来定位是哪个资源有错误，因为逻辑ID就是模板内的资源名称。

另外，当堆栈创建失败的时候，堆栈会自动回滚，将前面所有创建的资源都会删除，全都删除完成后，堆栈状态为ROLLBACK_COMPLETE，此时可以删除堆栈重新在创建一次。

2、CloudFormation模板进阶操作

2.1、模板资源属性

资源一共有6种属性，这里只介绍DependsOn属性。在两个资源之间有依赖关系时，你希望其中一个资源先创建，然后再创建另外一个资源，就可以用DependsOn属性。

例如你希望先创建数据库，然后再创建EC2实例，就可以在EC2资源下面加上DependsOn属性，确保创建完数据之后再创建EC2。

这里其实隐含了一个特性，就是AWS在创建、更新、删除资源的时候默认是并行的，也就是尽可能同时进行操作。使用DependsOn属性可以明确的指定资源创建的先后顺序，但是!Ref和!GetAtt 这两个内部函数具有隐式的依赖关系。前面的模板中，大量使用了!Ref函数，也可以确保被调用的资源先创建。

2.3、Parameters(参数)

Parameters是模板里面的可选字段。利用参数，你能够在每次创建或更新堆栈时将自定义值输入模板。参考文档

例如可以让用户自己选择EC2的密钥对，设置Web服务器的端口，后续安全组和输出都可以调用这个端口信息。

在安全组放行端口时，可以使用!Ref或者!Sub函数调用参数里面定义的Web服务器端口。

在创建堆栈时，需要填写参数信息。

2.4、Outputs(输出)

Outputs是模板里面的可选参数，你可以输出指定的值，这些值可以导入到其他堆栈中、从响应获取这些值、或者在AWS CloudFormation 控制台中查看。

例如这里输出EC2的公网IP地址，用!Sub和!Join两种方式输出Web服务器的URL。!Join函数这里简单演示一下使用案例，我个人更习惯使用!Sub函数。

2.5、Mappings(映像)

Mappings是模板里面的可选字段，包含一些键值对，通过Fn::FindInMap函数可以获取映射中值的信息。

Mappings字段在公开模板里面经常会看到。例如，因为AMI ID每个区域都是不一样的，所以想要模板内部的EC2实例能在多个区域运行，需要提前指定不同区域的AMI ID信息，此时就可以利用Mappings定义好不同区域的AMI ID信息，然后在EC2资源利用Fn::FindInMap调用。

这里定义了中国北京和中国宁夏区域的Amazon Linux AMI ID，HVM64表示全虚拟化x86架构64位系统，HVMG2表示Arm架构Graviton2处理器。参考文档

在指定EC2镜像时，可以使用!FindInMap函数进行调用。其中!Ref "AWS::Region"是伪参数，可以获取创建堆栈的AWS区域。这里就实现了在哪个区域部署时，就传递对应区域的AMI ID信息，模板将具有更好的灵活性。

另外一个常见使用Mappings的场景，是根据测试环境或者生产环境自动选择EC2实例类型，或者其他参数。

Mappings定义测试环境和生产环境的实例类型，并提供参数让用户选择环境类型。

EC2实例调用用户选择的环境类型，来决定启动的实例类型。

AWS CLI配置凭证

安装AWS CLI客户端之后，可以利用aws configure快速配置凭证。需要提前从AWS控制台创建对应账号的AK(Access Key)和SK(Secret Key)。

直接使用aws configure产生的是default配置文件，在没有明确指定配置文件时，就会使用default配置文件。你还可以使用aws configure --profile cn-prod来创建名称为cn-prod的配置文件。假设default配置的是测试账号信息，cn-prod配置的是生产账号信息，这样不明确指定配置文件的情况下，代码会使用测试环境的凭证。

AWS会将配置文件放在.aws文件夹中，这个文件夹默认在用户主目录下，例如Windows的路径C:\Users\crosswalk\.aws，Linux root用户路径/root/.aws。文件夹中会产生一个凭证配置文件credentials和一个Config配置文件config。
